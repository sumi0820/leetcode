### アプローチ 1

#### 何を考えて解いていたか

- 問題文を一読しただけだとどういう要件なのかが分からなかった
  - 重複文字ではない文字を探す
  - その中で最初の非重複文字のインデックスを返す
- 方針
  - 文字と出現回数を key/value ペアで Map する
  - value をもとに配列化し、最初の 1 のインデックスを検索する

```typescript
function firstUniqChar(s: string): number {
  const seen = new Map<string, number>();

  for (let i = 0; i < s.length; i++) {
    let val = seen.get(s[i]);
    seen.set(s[i], val ? val + 1 : 1);
  }

  const firstIndex = [...seen.values()].indexOf(1);
  return firstIndex;
}
```

function firstUniqChar(s: string): number {

### アプローチ 4

- findIndex を使用しない方法を検討

```typescript
function firstUniqChar(s: string): number {
  const seen = new Map<string, number>();

  for (const char of s) {
    seen.set(char, (seen.get(char) ?? 0) + 1);
  }

  for (const char of s) {
    if (seen.get(char) === 1) return s.indexOf(char);
  }
  return -1;
}
```

#### パフォーマンスを比較

- GPT でアプローチ３（findIndex）とアプローチ４（indexOf）のパフォーマンスを比較

##### パフォーマンス

| 指標               | `findIndex` | `indexOf`    | コメント                          |
| ------------------ | ----------- | ------------ | --------------------------------- |
| 時間計算量         | O(n)        | O(n²)        | 大規模入力では `findIndex` が有利 |
| 空間計算量         | O(n)        | O(1)         | 配列コピー分が `findIndex` に発生 |
| 実行パフォーマンス | 高速        | 低下しやすい | 定数倍でも `findIndex` が優勢     |

##### 実装特性

| 観点       | `findIndex`          | `indexOf`                    |
| ---------- | -------------------- | ---------------------------- |
| 記述量     | 宣言的に書ける       | ループ＋`indexOf` で冗長気味 |
| 可読性     | 処理意図が明確       | 追いかけがやや面倒           |
| メモリ効率 | 配列展開でコストあり | Map のみで軽い               |

##### 処理ステップ

| ステップ          | `findIndex` のコスト | `indexOf` のコスト |
| ----------------- | -------------------- | ------------------ |
| 1. Map でカウント | O(n)                 | O(n)               |
| 2. 文字列の走査   | `[...s]` で O(n)     | `for` で O(n)      |
| 3. 探索ロジック   | `findIndex` が O(n)  | `indexOf` が O(n²) |
| 合計時間計算量    | O(n)                 | O(n²)              |
| 空間計算量        | O(n)                 | O(1)               |

→ 大規模な入力では `findIndex` が一貫して高速で、`indexOf` はメモリ効率に利点がある。

## 気づいたこと/ memo

- 他の方のサロゲートペアに対応する実装を元に TypeScript に置き換える
- `codePoint > 0xffff ? 2 : 1`
  - GPT 指摘箇所
  - UTF-16 は 16 ビットしかないので、1 文字で表現できるのは 0xFFFF までのため、それを超える文字（例：絵文字）は 2 つの 16 ビット単位（サロゲートペア）で表される
  - つまり、16 ビットで表せる → 通常の 1 文字、サロゲートペア（2 ユニット）を使う絵文字など → 2 文字、とする

```typescript
function firstUniqChar(s: string): number {
  // Map<Unicodeコードポイント, 出現回数>
  const charToFoundCount = new Map<number, number>();

  // UTF-16を考慮して、コードポイント単位で走査
  for (let i = 0; i < s.length; ) {
    const codePoint = s.codePointAt(i)!;
    charToFoundCount.set(codePoint, (charToFoundCount.get(codePoint) ?? 0) + 1);
    i += codePoint > 0xffff ? 2 : 1; // サロゲートペアは2文字ぶん進める
  }

  // 2回目の走査で最初のユニーク文字を探す
  for (let i = 0; i < s.length; ) {
    const codePoint = s.codePointAt(i)!;
    if (charToFoundCount.get(codePoint) === 1) {
      return i;
    }
    i += codePoint > 0xffff ? 2 : 1;
  }

  return -1;
}
```

## 気づいたこと

- findIndex も indexOf も使用しないアプローチ
- charCodeAt()を使用して文字コードを利用してインデックスを計算する
  - `a`の文字コードは`97`なのでこれが基準
  - 例：）
    - 's'.charCodeAt(0) - base = 115 - 97 = 18
    - freq は 26 文字分の要素を 0 で持っており、`freq[18]++`していくことで 18 番目の文字のカウンターをインクリメントする

```typescript
function firstUniqChar(s: string): number {
  // 26文字（a〜z）ぶんの数値配列。.fill(0) で全要素を初期化
  const freq = new Array<number>(26).fill(0);

  // `a`の文字コード`97`を取得
  const baseCharCode = "a".charCodeAt(0);

  // 1パス目: 出現回数カウント
  for (let i = 0; i < s.length; i++) {
    freq[s.charCodeAt(i) - baseCharCode]++;
  }

  // 2パス目: 先頭の一意文字の位置を返す
  for (let i = 0; i < s.length; i++) {
    if (freq[s.charCodeAt(i) - baseCharCode] === 1) return i;
  }
  return -1;
}
```
